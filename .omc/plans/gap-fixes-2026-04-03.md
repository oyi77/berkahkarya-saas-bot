## RALPLAN-DR Summary

### Principles
1. **Credit integrity first** — every credit mutation is atomic and auditable; no silent gains or losses
2. **Zero trust on API keys** — key never stored in plaintext; checked at request time, not session time
3. **Fail closed on payment** — charge happens only after confirmed payment signal; refund on any ambiguity
4. **Smallest viable touch** — each gap fixed in isolation; no cross-gap refactors unless forced
5. **No feature flag debt** — wired but untriggered code is not acceptable; if a feature is advertised it must work

### Decision Drivers (top 3)
1. **Conversion funnel integrity** — welcome bonus is the first thing users expect; broken = lost users
2. **Trust / promise alignment** — Agency plan advertises "API Access" which does not exist; false claim must be resolved
3. **Revenue retention** — credit expiry without rollover is a measurable churn driver vs Pika/Kling

### Viable Options

#### Gap 3 (Agency API Auth) — three options considered

**Option A: Dedicated ApiKey model + middleware (chosen)**
- Pros: industry-standard; revocable per user; audit trail; no shared secret; fulfills the advertised feature
- Cons: schema migration + ~3h implementation; security-critical new surface

**Option B: Re-use admin token for agency users**
- Cons: shared secret — one leak exposes all users; no per-user revocation; not programmatically usable
- Rejected: violates Principle 2 ("Zero trust on API keys")

**Option C: Remove "API Access" claim from Agency plan description**
- Pros: zero implementation cost; zero security surface; honest
- Cons: degrades Agency tier value; requires messaging to existing agency subscribers; removes a competitive differentiator
- Rejected: We have no evidence of active agency subscribers who expected API access, but removing the claim reduces the tier's value proposition before we understand user demand. The safer path is to implement a minimal, secure API key system and make the claim true — this also enables future automation for power users. If demand proves zero after 90 days of tracking key generation, the feature can be removed.

#### Gap 5 (Lip Sync provider) — two options

**Option A: D-ID API (`api.d-id.com`) (chosen)**
- Pros: REST API, pay-as-you-go (~$0.02/5s), free tier for testing, supports image + text → talking video
- Cons: external dependency, polling model, new env var

**Option B: HeyGen API**
- Rejected: $29/mo floor regardless of usage; over-engineered for MVP; re-evaluate once demand is validated

## Credit Type Taxonomy (explicit)

| Type | Column | Affected by sub expiry? | Affected by rollover? |
|------|--------|------------------------|-----------------------|
| Subscription | `subscriptionCredits` tracks these | Yes (zeroed) | Yes (20% carry) |
| Purchased (topup) | `creditBalance - subscriptionCredits` | No | No |
| Welcome bonus | Treated as **purchased** | No | No |

**Implementation implication:** `grantWelcomeBonus` adds to `creditBalance` only. It does NOT touch `subscriptionCredits`. This means the bonus is never zeroed by sub expiry and is not subject to rollover, matching the intent of a "free permanent credit."

## Pre-Mortem (Deliberate Mode)

**Scenario 1: Welcome bonus double-grant on Telegram duplicate delivery**  
Telegram can deliver the same update twice. If `onboarding.ts` callback fires twice, the `$transaction` with `WHERE welcomeBonusUsed = false` guard is the only protection. If both transactions start simultaneously before either sets `welcomeBonusUsed = true`, PostgreSQL's row-level locking inside `$transaction` + serializable isolation will block the second — but only if Prisma uses `FOR UPDATE` or `serializable`. Default Prisma isolation is "read committed" — not sufficient.  
*Mitigation:* Use `update({ where: { telegramId, welcomeBonusUsed: false }, data: { welcomeBonusUsed: true, ... } })` as a single atomic CAS operation. Prisma's `update` with a predicate will throw `RecordNotFound` (P2025) if `welcomeBonusUsed` was already set — catch that error silently as "already granted."

**Scenario 2: Migration failure during active subscription renewal**  
Adding `ApiKey` model + `gatewayTransactionId` unique constraint requires two migrations. If the `gatewayTransactionId` unique constraint migration fails (due to existing duplicate nulls in the column), the table is left in a partially-altered state.  
*Mitigation:* Before running the migration, query `SELECT gateway_transaction_id, COUNT(*) FROM "Transaction" WHERE gateway_transaction_id IS NOT NULL GROUP BY gateway_transaction_id HAVING COUNT(*) > 1` to confirm no duplicates. Add `@@unique` only if zero duplicates found.

**Scenario 3: Stars `successful_payment` fires, credit grant fails, event lost**  
If `UserService.addCredits()` throws after Telegram fires `successful_payment`, the user paid but received no credits. Telegram does NOT retry this event.  
*Mitigation:* Write Transaction record FIRST (idempotency key = `telegram_payment_charge_id`). If credit grant fails, the Transaction record proves payment occurred. Add admin dashboard query to find Transactions with `status='success'`, `gateway='stars'` where the user's credit was not updated — enables manual recovery.

- [ ] New user completing `/start` → language select → `creditBalance = 1.0` in DB
- [ ] `welcomeBonusUsed = true` set in same atomic update; calling the grant function twice results in second call being silently no-op (P2025 caught)
- [ ] `Transaction` row written with `type = 'welcome_bonus'`, `gateway = 'system'`, `status = 'success'`, `creditsAmount = 1`
- [ ] Telegram message sent to user confirming 1 credit received
- [ ] Existing user re-running `/start` receives no additional credits

### Gap 2 — Credit Rollover
- [ ] User with `subscriptionCredits = 20`, `creditBalance = 8` on a 20-credit plan → renewal gives `new_credits + floor(min(20×0.2, 8×0.2)) = new_credits + floor(min(4, 1.6)) = new_credits + 1` (integer floor of 1.6 = 1)
- [ ] User with `subscriptionCredits = 20`, `creditBalance = 20` (used nothing) → rollover = `floor(20 × 0.2) = 4`
- [ ] User with `subscriptionCredits = 0`, `creditBalance = 5` (all purchased) → rollover = 0, purchased credits untouched
- [ ] `expireStaleCredits()`: user with `subscriptionCredits = 10`, `creditBalance = 7` (spent 3 purchased) → after expiry: `creditBalance = Math.max(0, 7 - 10) = 0` (sub credits gone, no negative balance)

**Step 2 — Middleware (`src/middleware/api-auth.ts`)**
```ts
import crypto from 'crypto';
import { prisma } from '@/config/database';
import type { FastifyRequest, FastifyReply } from 'fastify';

export async function apiKeyAuth(request: FastifyRequest, reply: FastifyReply) {
  const raw = request.headers['x-api-key'] as string | undefined;
  if (!raw) return reply.status(401).send({ error: 'Missing X-API-Key header' });
  const hash = crypto.createHash('sha256').update(raw).digest('hex');
  const apiKey = await prisma.apiKey.findUnique({
    where: { keyHash: hash },
    include: { user: { select: { telegramId: true, tier: true } } }
  });
  if (!apiKey || apiKey.revokedAt) return reply.status(401).send({ error: 'Invalid or revoked key' });
  if (apiKey.user.tier !== 'agency') return reply.status(403).send({ error: 'Agency tier required' });
  // Batch lastUsedAt: only update if > 1 min since last update (avoid write-per-request)
  const oneMinAgo = new Date(Date.now() - 60_000);
  if (!apiKey.lastUsedAt || apiKey.lastUsedAt < oneMinAgo) {
    await prisma.apiKey.update({ where: { id: apiKey.id }, data: { lastUsedAt: new Date() } });
  }
  (request as any).apiUser = apiKey.user;
}
```

  const apiKey = process.env.D_ID_API_KEY;

## Risks & Mitigations

| Risk | Severity | Mitigation |
|---|---|---|
| Welcome bonus double-grant on Telegram duplicate delivery | HIGH | Atomic CAS via `update({ where: { telegramId, welcomeBonusUsed: false } })`; catch P2025 silently |
| Rollover race condition on concurrent credit deduction | HIGH | Entire `renewSubscription()` wrapped in `$transaction` |
| `gatewayTransactionId` unique migration fails on existing duplicates | HIGH | Pre-migration SQL check for duplicates; migration blocked if any found |
| Stars `successful_payment` lost after credit grant fails | HIGH | Write Transaction BEFORE credit grant; catch P2002 for dedup; admin recovery query available |
| API key raw value exposure in logs | HIGH | Never log raw key; store only SHA-256 hash; plaintext returned once to client, never persisted |
| D-ID service outage blocks talking photo | MEDIUM | 90s timeout + refund; circuit breaker if 3+ consecutive failures |
| `expireStaleCredits` produces negative balance | MEDIUM | `Math.max(0, creditBalance - subscriptionCredits)` guard |
| Rollover math rounds down aggressively (integer floor) | LOW | Accepted: integer credits are simpler and prevent fractional accumulation edge cases |

## ADR

**Decision:** Fix 5 gaps with targeted changes, no architectural overhaul.

**Drivers:**
1. Conversion funnel broken (welcome bonus)
2. False claim on paid plan (Agency API Access)
3. Churn driver (credit expiry without rollover)

**Alternatives considered:**
- Full credit ledger refactor (event sourcing): rejected — 3-5x more scope, not needed for these fixes
- Remove "API Access" claim from Agency: rejected — cheaper but reduces tier value before demand is measured
- HeyGen for lip sync: rejected — fixed cost floor, better after demand validation

**Why chosen:** Isolated fixes on confirmed gaps; each change is reversible except schema migrations (which have rollback SQL); total scope fits within 10h.

**Consequences:** 
- Two schema migrations required (ApiKey model, gatewayTransactionId unique)
- D-ID introduces a new external dependency
- Agency API surface is new security-critical code requiring review

**Follow-ups:**
- Monitor API key generation rate for 90 days — if zero usage, reconsider feature
- Add D-ID to circuit breaker service after 30 days of production data
- Audit all credit mutation paths to confirm `subscriptionCredits` is updated correctly everywhere
